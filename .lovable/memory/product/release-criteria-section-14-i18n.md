# §14 — Internationalization & Localization

Goal: ship EN-only at launch **without** painting ourselves into a corner. Every user-facing string is already routed through an i18n wrapper (per §7 a11y + Core memory), and the data layer is locale-aware so adding a second language later is a translation job, not a refactor.

## Scope at launch
- **One locale shipped**: `en-US`. No language switcher visible.
- **Wrapper in place everywhere**: every user-facing string (UI chrome, toasts, errors, emails, push copy, AI system prompts that produce user-visible output) goes through `t("namespace.key")`. No bare string literals in JSX/TSX outside of:
  - Brand wordmarks (DYNAMO, SUPERCELL-ID)
  - User-generated content (transcripts, notebook bodies, profile bios)
  - Format/topic names entered by users
- **Lint rule**: ESLint custom rule `no-literal-jsx-strings` (warn at launch, error post-launch) to keep new code wrapper-compliant.
- **Source-of-truth file**: `src/i18n/en-US.json`, flat key namespace per surface (`home.*`, `debate.*`, `live.*`, `record.*`, `auth.*`, `errors.*`, `push.*`, `email.*`, `dynamo.*`).
- **Loader**: lightweight in-house (`src/i18n/index.ts`) — no `i18next` runtime at launch (saves ~30KB). Build-time bundle of the active locale; `useT()` hook returns `t(key, vars?)` with `{var}` interpolation + ICU-style plural via simple `plural(count, {one, other})` helper.

## Data-layer locale-readiness
- **User preference column**: `profiles.locale text not null default 'en-US'`. Set at signup from `navigator.language` (clamped to supported list — at launch, only `en-US`, so always `en-US`).
- **Session locale**: `live_sessions.locale`, `debates.locale`, `cmm_sessions.locale` — defaults to host's `profiles.locale`. Determines which language the AI Facilitator (DYNAMO) responds in + which Deepgram model is selected.
- **Per-message locale**: transcripts already store raw text; we add `transcript_segments.detected_lang text` (nullable) populated by Deepgram's detection. Not displayed at launch; recorded for future multilingual sessions.
- **Email/push templates**: keyed by `(template_id, locale)`. At launch, only `en-US` rows exist; renderer falls back to `en-US` if a requested locale is missing.

## Date / number / currency formatting
- All date formatting goes through `src/lib/date.ts` — already exists; extend to read `profiles.locale` rather than hardcoding `en-US`.
- Numbers via `Intl.NumberFormat(locale)`.
- Currency (Stripe — §12) always rendered with `Intl.NumberFormat(locale, { style: 'currency', currency: priceCurrency })` so a future EUR price displays correctly.
- Relative time ("2h ago") via `Intl.RelativeTimeFormat`.

## AI / transcription locale handling
- **Deepgram**: model selection by `session.locale`. At launch, `nova-2-general` (EN). Locale → model map lives in `supabase/functions/deepgram-token/index.ts`.
- **DYNAMO (AI Facilitator)**: system prompt includes `"Respond in {locale} only."`. Launch prompts are English; the substitution is in place so a future Spanish session swaps cleanly.
- **Analyze-transcript / record-qa edge functions**: pass `session.locale` into the prompt; quote attribution stays in source language (we never auto-translate transcripts).

## RTL / bidi
- Tailwind `dir="ltr"` hardcoded on `<html>` at launch.
- No RTL-specific CSS work shipped. Logical properties (`ms-`, `me-`, `ps-`, `pe-`) **encouraged** in new code so a future `dir="rtl"` flip is mechanical, but not enforced.

## Translation pipeline (post-launch, but architected now)
- Source file `en-US.json` is the canonical key list.
- Future locales are sibling files (`es-419.json`, `pt-BR.json`, etc.) generated by an external TMS (e.g., Crowdin) or in-house with Lovable AI.
- CI check (post-launch): missing keys in non-source locales → warning, not blocking; the loader falls back to `en-US` for missing keys at runtime.

## Out of scope at launch
- Visible language switcher.
- Any locale other than `en-US`.
- RTL layout / logical-property migration of legacy components.
- Translating user-generated content (transcripts, notebooks, profiles, club descriptions).
- Translating AI-generated summaries into a different language than the source session.
- Per-region pricing / VAT (Stripe handles tax; pricing is single-currency USD at launch — see §12/§18).
- Localized SEO routes (`/es/...`).
- Locale-specific moderation rules.

## KPIs (PostHog)
`i18n_missing_key` (key, locale) — fires when loader falls back; should be ~0 at launch.
`locale_detected_at_signup` (locale string from `navigator.language`, even though clamped) — informs which locales to prioritize post-launch.
`session_locale_set` (locale) — for future multilingual analytics.

## Acceptance checklist
- [ ] Zero bare user-facing string literals in `src/pages/**` and `src/components/**` (lint passes).
- [ ] `t("nonexistent.key")` returns the key string (not crash) and emits `i18n_missing_key`.
- [ ] `profiles.locale` populated for new signups from `navigator.language` (clamped to `en-US`).
- [ ] `live_sessions.locale` / `debates.locale` / `cmm_sessions.locale` default to host locale on insert.
- [ ] Date / time / number / currency display via `Intl.*` with the user's locale, not hardcoded `en-US`.
- [ ] Deepgram token endpoint selects model based on `session.locale`.
- [ ] DYNAMO system prompt includes `"Respond in {locale} only."` with substitution working.
- [ ] Email + push templates resolved via `(template_id, locale)` lookup with `en-US` fallback.
- [ ] Adding a second locale file ships translated UI without code changes (smoke-tested with a 10-key `es-419.json` stub behind a dev flag).
